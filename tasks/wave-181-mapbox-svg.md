# wave/181 — Rasterize MTN coverage placeholder (P2.1)

## Symptom

Browser console on prod (`farm-management-lilac.vercel.app`) emits ~141
identical errors per page load on any view that mounts `<FarmMap>`:

```
Error: Could not load image because of The source image could not be decoded.
       Image source: data:image/svg+xml;utf8,…
       Note that SVGs are not supported.
```

Origin chunk: `ab749c57.f91995e7178866fd.js:1:23837` (the bundled
`MtnCoverageLayer.tsx`).

Visible side-effects:

1. The MTN coverage overlay never appears, even when the user enables it
   in `LayerToggle` — silent UX failure.
2. CSP report sink for Wave 4 A8 (PR #77) would receive ~141 violation
   reports per session once the **2026-05-11 enforce flip** lands, since
   each failed image decode counts as a CSP `img-src` runtime fetch
   (`data:` is on the connect/img allowlist but Mapbox surfaces these as
   page-level errors regardless).

## Root cause

`components/map/layers/MtnCoverageLayer.tsx` registered an inline SVG
data URL (`data:image/svg+xml;utf8,…`) as the tile source for a Mapbox
`raster` source. **Mapbox GL JS raster sources only decode PNG, JPG, or
WebP**; they explicitly do not support SVG (per Mapbox source:
`src/util/image.ts → isImageBitmap`). Mapbox calls
`fetch(tileUrl) → createImageBitmap(blob)`, and `createImageBitmap`
throws "SVGs are not supported" for `image/svg+xml` blobs. The error
fires once per requested tile (typical farm-map viewport at zoom 12-15
across ~3 zoom levels = ~141 tiles).

The bug shipped in **Wave 2D** as a developer convenience: the SVG was
intended as a "wiring visible without a real upstream" placeholder.
Author knew Mapbox would receive the data URL but did not know SVG was
unsupported in the raster decode path (it works fine in HTML
`<img>`/`background-image`, hence the local-dev confusion).

out of scope.

## Why PNG (not Canvas-rasterized SVG, not switching to Marker)

- **Canvas rasterize at runtime**: works but adds a `<canvas>` +
  `Image()` + `toDataURL()` round-trip on first mount. Net cost:
  ~10ms; net benefit: code stays "feels like SVG". Not worth the
  fragility (requires `typeof document !== "undefined"` guard and a
  `useMemo` dance).
- **Switch to `<Marker element={svgEl} />`**: doesn't apply here. This
  is a tiled raster overlay, not a per-feature pin. Marker is for
  point geometries with DOM elements.
- **Pre-rasterized PNG**: 114 bytes inline, deterministic, no runtime
  cost, no SSR concerns, decodes correctly under Mapbox's
  `createImageBitmap` path. The PNG was generated by a one-shot
  `node` script (zlib + minimal PNG encoder, 32×32, 16×16 amber
  squares at 12% alpha — visually equivalent to the prior SVG).

   (replaces what was previously a transparent / nothing rendered).

### Regression guard

The test in `__tests__/components/map/MtnCoverageLayer.test.tsx` will
fail on any future diff that flips the placeholder back to SVG (or
strips the PNG header). CI catches the regression before it ships.
